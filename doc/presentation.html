Slide 02

1. 플랫폼 도입 배경 및 목표

✦ 고객 관리 체계 고도화

기업 고객 정보의 안전한 등록 및 관리와 함께, 임직원 권한에 기반한 정보 숨김 제어 기능을 지원하여 중요 기업 정보의 보안 노출 리스크를 완전히 통제합니다.

✦ 고객별 소프트웨어 제품 관리

대항목 및 세부 항목 옵션 관리 체계 구축과 고객별 소프트웨어 설치 이력(제품이름, 설치 버전, 설치 OS, 업그레이드 참고사항 및 이력 타임라인 관리)의 데이터 정합성을 제공합니다.

Slide 03

2. 시스템 아키텍처 구성

Frontend (UI)

HTML5와 Vanilla JS로 구축되어 백엔드 API만 호출(CORS 허용). 사이드바 레이아웃 공통 모듈화 및 Midnight Aurora CSS 테마 탑재.

Backend (API)

PHP 8.2 기반의 RESTful API 아키텍처. 요청 규격을 공통화(Response)하고 세션 기반 권한 체크 필터를 통과한 데이터만 처리.

DB Layer (SQL)

MySQL 2-Tier 릴레이션 설계. 테이블 단위 쿼리 라이브러리(SQL Class)를 격리 구현하여 비즈니스 로직의 의존성을 완화.

CMS/ ├── openapi/ # API YAML 명세서 (v1 동기화) │ ├── member.yaml │ ├── customer.yaml │ ├── product.yaml │ ├── mng_category.yaml │ └── mng_item.yaml ├── backend/ # RESTful API 엔드포인트 │ ├── v1/ # 1차 Release v1 프록시 레이어 │ ├── member/ # 가입, 로그인, 목록 제어 │ ├── customer/ # 고객 목록 및 상세 정보 │ ├── product/ # 버전/OS 변경 이력 및 업그레이드 │ └── mng_item/ # 세부 옵션 CRUD ├── frontend/ # SPA 화면 및 스타일시트 │ ├── common/ # style.css, api.js, layout.js │ ├── category/ # 대항목 및 세부 코드 관리 화면 │ └── customer/ # 고객 상세 및 제품 리스트 └── lib_sql/ # PDO기반 테이블 쿼리 클래스
  • openapi/: FE/BE 개발 약속이 정의된 YAML 명세서 수록. (v1 서버 및 대항목/점검 추가 완료)

  • frontend/: 카테고리 관리 화면 신설 및 Midnight Aurora 디자인 기반 UI 영역.
  • lib_sql/: 카테고리/세부항목/이력관리 전용 SQL 쿼리 라이브러리 추가 탑재.

    4. 관계형 데이터베이스 스키마

    테이블명 역할 설명 주요 핵심 컬럼 구성 관계 및 매핑 규칙
    member 임직원 계정 테이블 id, login_id, password, role, created_at 관리자(admin)와 사원(user) 역할 구분
    customer 고객사 정보 테이블 id, company_name, description, is_hidden description(기타사항) 수록, 일반계정 숨김지원
    product 설치 제품 테이블 id, customer_id, name, model_name, version, os_type name(제품이름) 컬럼 추가, 하드웨어 연동 제거
    mng_category 대항목 카테고리 id, code (HW/MAGUX_VER/OS), name, description 제품 옵션의 상위 카테고리 (1 : N mng_item)
    product_history 제품 업그레이드 이력 id, product_id, old_version_code, new_version_code, notes 제품 스펙 업그레이드 이력 및 참고사항 타임라인 로깅

    5. API 설계 및 보안 메커니즘

    • 역할 기반 접근 제어 (RBAC): 회원 등록, 회원 권한 제어, 하드웨어 스펙 추가, 고객사 숨김 스위칭 등 민감한 동작은 Auth::requireAdmin()을 통과한 세션에 한해서만 수행됩니다.
    • 입력 유효성 검사 강화: ID 중복 검사, 패스워드 6자 유효성, 이메일 규격 등을 백엔드 API에서 이중으로 실시간 유효성 체크합니다.
    • 보안성 강화: 세션 고정 공격 방지를 위한 session_regenerate_id() 및 패스워드 bcrypt 일방향 해시 암호화 기본 탑재.

    6. OpenAPI 기반 분리 아키텍처 효과

    ✦ 약속 기반 독립 개발 (Loose Coupling)

    API 규약(OpenAPI YAML)을 먼저 선언 후 프론트엔드/백엔드를 분리 개발하여, 상호 코드 간섭과 결합성을 배제하고 병렬 생산성을 극대화했습니다.

    ✦ 이기종 플랫폼 확장성 확보

    백엔드가 순수 JSON API로 작동하므로, 웹 프론트엔드 코드의 변경 없이 즉시 모바일 앱(iOS, Android)이나 외부 시스템과의 데이터 연동이 가능합니다.

    ✦ 서버 부하 절감 및 보안 격리

    렌더링 연산을 브라우저가 전담하여 서버 오버헤드가 크게 줄었으며, DB 접속 정보를 프론트엔드에 노출하지 않아 물리적 보안 장벽을 형성했습니다.

    ✦ 명확한 이슈 격리 (Debugging)

    데이터 전송 문제와 화면 바인딩 문제가 크롬 개발자 도구의 Network 탭을 통해 즉각 파악되어 테스트 리드타임이 대폭 감소했습니다.

    7. OpenAPI 3.0 기반 협업 워크플로우

    OpenAPI 3.0

    단일 진실 공급원
    (Single Source of Truth)

    ✦ 기획자 (Planner)

    비즈니스 요구사항을 수립하여 OpenAPI 명세 정의. 기획 의도대로 화면과 로직이 개발되도록 설계의 표준을 세우고 개발자들과 공유합니다.

    ✦ Frontend 개발자

    중앙의 OpenAPI 규격을 참조하여 백엔드 연동 없이 Mock API 기반 UI 단독 개발. 병렬 개발을 통해 시간 지연 없이 UX 화면을 선제 완성합니다.

    ✦ Backend 개발자

    명세의 입력/출력 타입에 매칭되는 RESTful API 구현. 명세에 따른 세션 인증 필터 및 DB 트랜잭션을 구현하여 규격을 엄격히 보장합니다.

    Slide 09

    8. 기대 효과 및 결론

    ✦ 뛰어난 유지보수 구조

    프론트엔드 HTML과 백엔드 PHP API의 완벽한 2-Tier 격리로, 향후 클라이언트 프레임워크 전환이나 DB 튜닝 시 영향도를 최소화합니다.

    ✦ 독보적인 사용자 경험 (UX)

    Midnight Aurora 스펙에 부합하는 오로라 그라데이션 및 네온 라이팅 효과를 디자인 시스템 수준으로 코딩하여, 기능의 완성도와 심미성을 결합했습니다.
